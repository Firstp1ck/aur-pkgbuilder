# Design specification for an AUR maintainer GUI, end to end

A GUI that genuinely shepherds a maintainer from a fresh `~/.ssh` directory to a live AUR package must treat the workflow as eleven tightly coupled stages, not as an editor with a "push" button. The single most important insight is that **AUR is an unmoderated git-over-SSH service with strict server-side hooks**: every commit must contain both a valid `PKGBUILD` and a synchronized `.SRCINFO`, the branch must be `master`, and no force-pushes are allowed. Everything else — SSH setup, build hygiene, namcap linting, nvchecker monitoring — exists to make those hook checks pass reliably and to produce packages that actually work on clean systems. This specification walks through each stage with the exact commands, error strings, state shape, and UX affordances a desktop GUI should implement, flags the subtle requirements that are most frequently missed, and closes with GUI-wide architectural recommendations.

## SSH setup and AUR authentication

The application begins where most tutorials begin: `~/.ssh`. On first launch the GUI should scan that directory for the OpenSSH default identity filenames (`id_ed25519`, `id_ed25519_sk`, `id_ecdsa`, `id_ecdsa_sk`, `id_rsa`, and the legacy `id_dsa`) plus an AUR-specific convention of `~/.ssh/aur` that the ArchWiki recommends. For every `*.pub` found, it should shell out to `ssh-keygen -lf <path>` to render a SHA256 fingerprint and the key type so the user can identify each key in a list view. Detecting existing keys is important not because the GUI should reuse them — the ArchWiki explicitly recommends a dedicated AUR key so that revoking it affects nothing else — but because the user needs to see, not guess, what is on disk before the wizard offers to create another one.

The canonical generator for a new AUR identity is `ssh-keygen -t ed25519 -f ~/.ssh/aur -C "aur@$(uname -n)"`. The GUI should default to Ed25519 (256-bit, fixed; `-b` is ignored), offer ECDSA (256/384/521) and RSA (4096 recommended, 3072 minimum useful) as fallbacks, and warn the user if they pick RSA below 3072 or attempt DSA, which OpenSSH 10.0+ has removed and the AUR server-side `valid-keytypes` list technically still accepts as `ssh-dss` but which modern clients cannot use. **A passphrase is strongly recommended**: without one, possession of the key file is equivalent to "can push anything to any of your AUR packages." The GUI should offer a first-class "add to agent" step immediately after generation — running `eval "$(ssh-agent -s)"` in the user's shell profile is brittle, so a better UX is to launch `ssh-add ~/.ssh/aur` and verify via `ssh-add -l` that the fingerprint appears, then let the user opt into integration with `gnome-keyring`, `kwallet`, or `keychain`. File permissions must be enforced: `~/.ssh` at 700, private keys at 600, public keys and `config` at 600 or 644 as appropriate. Anything looser causes OpenSSH to refuse the key with `Permissions 0644 for '…' are too open`, which is a failure mode the GUI must recognize and auto-correct with a confirmation prompt.

Uploading the public key is currently a browser-only operation: the AUR does not expose an endpoint to submit a key programmatically. The GUI should therefore present the contents of `~/.ssh/aur.pub` in a copy-to-clipboard widget, open `https://aur.archlinux.org/` to the "My Account" form in the default browser via `xdg-open`, and guide the user to paste into the "SSH Public Key" multiline textarea. **Multiple keys are supported by placing one per line** in that field — a crucial detail for users who rotate machines. A large share of "The SSH public key is invalid" errors on the Arch BBS trace to paste-induced stray whitespace or multi-line wrapping; the GUI should normalize its clipboard payload to a single trimmed line of `keytype base64 comment` form before handing it to the user. The GUI must note that any given public key can be attached to exactly one AUR account — aurweb enforces uniqueness — and that new account registration itself requires manual approval by emailing `accountsupport@archlinux.org` because of ongoing spam pressure, a fact worth surfacing prominently in the first-run wizard so users do not assume their initial submission disappeared.

The SSH client configuration step is small but error-prone. The GUI should append (or offer to append) a `Host aur.archlinux.org` stanza to `~/.ssh/config` specifying `Hostname aur.archlinux.org`, `User aur`, `IdentityFile ~/.ssh/aur`, and critically `IdentitiesOnly yes`. The last directive is the fix for the extremely common `Too many authentication failures` complaint: when a user has many keys loaded in ssh-agent (work GitHub, a personal GitLab, etc.), OpenSSH will offer them in succession and can exceed the server's attempt limit before reaching the correct one. Setting `IdentitiesOnly yes` forces the client to use only the explicitly named `IdentityFile` for this host. For robots and headless setups, `StrictHostKeyChecking accept-new` is a reasonable default because it rejects changed keys but accepts first-seen ones; for interactive setups the GUI should instead pre-seed `~/.ssh/known_hosts` itself.

Host fingerprint verification is where most tooling cuts corners. The GUI must not accept the host key on trust. The three currently-published SHA256 fingerprints for `aur.archlinux.org` are **Ed25519 `SHA256:RFzBCUItH9LZS0cKB5UE6ceAYhBD5C8GeOBip8Z11+4`**, **ECDSA `SHA256:uTa/0PndEgPZTf76e1DFqXKJEXKsn7m9ivhLQtzGOCI`**, and **RSA `SHA256:5s5cIyReIfNNVGRFdDbe3hdYiI5OelHGpw2rOUud3Q8`**, displayed on the logged-out AUR homepage and in the ArchWiki. The correct flow is to run `ssh-keyscan aur.archlinux.org` at setup time, compute `ssh-keygen -lf` on the resulting key file, and compare each line against the three known fingerprints *as fetched live from `https://aur.archlinux.org/` and the ArchWiki at runtime*. Hard-coding the values is acceptable as a fallback but the tool should have a refresh path because AUR can and will eventually rotate keys, in which case users will otherwise see `REMOTE HOST IDENTIFICATION HAS CHANGED` and have to resolve it manually. Connectivity should be confirmed with `ssh aur@aur.archlinux.org help`, which on success prints `Welcome to AUR, <username>!` followed by a disabled-interactive-shell notice and exits with code 1 — the non-zero exit is expected because aurweb's git-shell refuses interactive sessions. The GUI should parse for the literal `Welcome to AUR` string rather than the exit code. Common failure strings to translate into actionable help are `Permission denied (publickey).` (key not uploaded, or `User aur` missing, or `IdentitiesOnly` missing), `Host key verification failed.` (stale `known_hosts` entry — offer `ssh-keygen -R aur.archlinux.org` and re-verify), and `Connection refused / timed out` (corporate firewall blocking 22 — suggest a proxy or alternate network; the AUR does not offer HTTPS push).

## Account and repository bootstrap

Once SSH works, the GUI becomes a git front-end with AUR-specific rules. Cloning an existing package is `git clone ssh://aur@aur.archlinux.org/<pkgbase>.git`, where `<pkgbase>` is the repository name — **never** `<pkgname>`, because for split packages these diverge. For a non-split package `pkgbase` defaults to `pkgname` and is not declared; for a split package `pkgbase` is either explicitly assigned or defaults to `pkgname[0]`. The AUR repository regex is `^[a-z0-9][a-z0-9.+_-]*$`: lowercase start, no leading hyphen or dot, and the server will reject `invalid pkgbase` on any violation — a non-obvious cause of failed first pushes for users who tried `CoreFreq` or `MyPkg`.

Creating a new package uses the same clone command against a non-existent pkgbase. Git emits `warning: You appear to have cloned an empty repository.` — **this is expected**, not an error. Aurweb's SSH-side service creates an empty on-demand namespace for any authenticated clone; it only becomes a real, web-visible package after a successful first push commits a valid tree. The GUI should present this warning as informational with text like "Empty namespace ready. Add PKGBUILD and .SRCINFO, then push to publish." The alternative of running `git init` locally and adding the remote by hand works but skips an implicit credential check, so the canonical clone-first pattern should be preferred and recommended by the UI. The GUI should suggest a working-directory layout of one clone per package under `~/aur/<pkgbase>/`, avoid submodules or monorepos by default, and offer an advanced "aurpublish-style monorepo" mode for power users who already organize around `git subtree split`.

It is worth noting up front that the AUR is **unmoderated**: there is no Trusted User review before a package goes live. The moment `git push` succeeds the package is public, and the maintainer bears sole responsibility for quality. Trusted Users — renamed Package Maintainers after RFC 0014 — handle orphan/deletion/merge requests and may delete rule-violating packages after the fact, but they do not gate submission. The GUI should reflect this in its pre-push checklist UX: the push button is the point of commitment, not a review request.

## PKGBUILD authoring

The PKGBUILD is a bash script with a strict set of recognized identifiers, and the GUI should expose it through a dual-view editor: a structured form for every known field with validation and a raw-text view for the functions and anything unusual. The mandatory fields are `pkgname` (string or array, lowercase alphanumerics plus `@ . _ + -`, never starting with `-` or `.`), `pkgver` (no colons, hyphens, slashes, or whitespace — upstream hyphens must become underscores), `pkgrel` (positive integer, starts at 1, reset on every `pkgver` change), `arch` (members of `any`, `x86_64`, `i686`, `pentium4`, `armv6h`, `armv7h`, `aarch64`), and a `package()` function. Fields that should be treated as effectively mandatory by the linter are `pkgdesc` (one line, conventionally ≤ 80 characters, and by convention without the self-referential "Foo is a ..." opening), `url`, and `license`. Optional but common are `epoch`, `groups`, `depends`, `makedepends`, `checkdepends`, `optdepends` (values formatted as `'pkg: reason'`), `provides`, `conflicts`, `replaces`, `backup`, `options`, `install`, `changelog`, `source`, `noextract`, `validpgpkeys`, one of the integrity arrays, and the functions `verify`, `prepare`, `pkgver`, `build`, `check`, and `package`. All dependency arrays and `source`/`sums` arrays accept architecture suffixes — `depends_x86_64=()`, `source_aarch64=()`, `sha256sums_x86_64=()` — and the GUI must propagate these suffixes cleanly because they are lost constantly when maintainers hand-edit.

Licensing is the single largest churn point for 2024–2026. Under RFC 16 the `license=()` array must hold **SPDX identifiers** (`GPL-3.0-or-later`, `Apache-2.0`, `MIT`, `BSD-3-Clause`, and composite expressions with `AND`, `OR`, `WITH`, and `+`). Custom licenses use `LicenseRef-<name>` or `custom:<name>` but both require shipping the text to `/usr/share/licenses/$pkgname/LICENSE`. Bare legacy tags — `GPL`, `LGPL`, `BSD`, `custom` — are deprecated, still transitionally accepted, and must be flagged by the GUI with a one-click "convert to SPDX" suggestion drawing from the `licenses` package's shipped SPDX table under `/usr/share/licenses/spdx/`. Under RFC 40 the packaging sources themselves (PKGBUILD, install scripts, patches, systemd units) are licensed 0BSD, and under RFC 52 official repos use REUSE (`REUSE.toml` alongside PKGBUILD and a `LICENSES/0BSD.txt`). The GUI should offer a "set up REUSE" action that wraps `pkgctl license setup` and a compliance check wrapping `pkgctl license check` (which invokes `reuse lint`). For AUR packages these steps are recommended for packages the maintainer hopes to promote to `[extra]` but are not enforced by the AUR server.

The GUI's form editor should know the difference between global and per-split fields. For split packages, `pkgname` becomes an array (`pkgname=(foo foo-docs foo-devel)`) and each subpackage gets a `package_<name>()` function that may override `pkgdesc`, `arch`, `url`, `license`, `groups`, `depends`, `optdepends`, `provides`, `conflicts`, `replaces`, `backup`, `options`, `install`, and `changelog`. Global `depends` and `makedepends` are used for `--syncdeps` resolution, so per-split `depends` alone will not install the right set at build time — the GUI should warn when a per-split `depends` entry is not transitively represented globally. Dependency entries accept version constraints (`qt6-base>=6.5.0`) and soname autodetection when the entry ends in `.so`; the form should offer pacman-db-driven autocomplete by scanning `pacman -Sl` and `pacman -Si` output, and it should query the AUR RPC for dependencies not found in official repositories so the user sees "found in AUR: yes/no" next to each entry in real time.

Source handling deserves special attention because this is where most PKGBUILDs get broken. The `source=()` array supports bare URLs, local files alongside the PKGBUILD, renamed fetches using `localname::url`, and VCS fetches prefixed `git+`, `svn+`, `hg+`, `bzr+`, or `fossil+` with fragments in the URL (`#branch=`, `#tag=`, `#commit=`, `#revision=`) and a `?signed` query for git-tag PGP verification against `validpgpkeys`. The GUI should render the array as a table with columns for the fragment type, the target name, the URL, and the verification mode, validating fragment keys against their VCS and complaining if a `-git`-named package lacks a `pkgver()` function or if a non-VCS source has `SKIP` as its checksum. Integrity arrays must have exactly one entry per `source` entry; mismatched lengths are a makepkg abort (`Integrity checks (...) differ in size from source array`). `sha256sums` is the effective minimum Arch standard in 2026 and what `makepkg -g` emits by default; `b2sums` is valid and faster but less common; `md5sums` and `sha1sums` are legacy and should be flagged as weak. For VCS sources the correct value is the literal string `SKIP`, and for PGP-signed tarballs the pattern is two entries — the real hash of the tarball and `SKIP` for the `.sig` — with a `validpgpkeys=()` entry containing full 40-character uppercase fingerprints.

The GUI should ship PKGBUILD templates keyed to the common archetypes. The standard source-tarball/autotools template wires `./configure --prefix=/usr` in `build()` and `make DESTDIR="$pkgdir/" install` in `package()`. The binary-package `-bin` template omits `build()`, uses `options=(!strip !debug)` because the upstream artifacts should not be modified, `install -Dm755` the shipped binary, and declares `provides=("${_pkgname}=${pkgver}") conflicts=("${_pkgname}")`. The git-VCS `-git` template sets `source=("$_pkgname::git+$url.git")` with `SKIP` checksum, defines `pkgver()` to run `git describe --long --tags --abbrev=7 | sed 's/^v//;s/\([^-]*-g\)/r\1/;s/-/./g'` inside the clone, and again provides/conflicts against the release variant. The Python template uses `python -m build --wheel --no-isolation` in `build()` and `python -m installer --destdir="$pkgdir" dist/*.whl` in `package()`, listing `python-build`, `python-installer`, `python-wheel`, and the project's PEP 517 backend (`python-hatchling`, `python-setuptools`, `python-flit-core`, or `python-poetry-core`) in `makedepends`. The Rust template lists `cargo` (not `rust` or `rustup`) in `makedepends`, runs `cargo fetch --locked --target "$(rustc -vV | sed -n 's/host: //p')"` in `prepare()`, `cargo build --frozen --release --all-features` in `build()`, and `cargo install --no-track --frozen --root "$pkgdir/usr" --path .` in `package()`; for offline chroot builds it sets `CARGO_HOME=$srcdir/.cargo` to vendor under `$srcdir`. The Go template exports `GOFLAGS="-buildmode=pie -trimpath -ldflags=-linkmode=external -mod=readonly -modcacherw"` and bridges `CGO_*FLAGS` from the caller's `*FLAGS` so hardening survives — and the GUI should warn that `-trimpath` is incompatible with the current debug-package machinery, forcing `options=(!debug)`.

The `pkgver()` function for VCS packages is subtle enough to deserve its own snippet library: annotated tags use `git describe --long --abbrev=7 | sed 's/\([^-]*-g\)/r\1/;s/-/./g'` producing `2.0.r6.ga17a017`; projects using unannotated tags need `--tags`; projects with a name-prefixed tag (`foo-1.0`) require `sed 's/^foo-//;...'`; repos without tags fall back to `printf "r%s.%s" "$(git rev-list --count HEAD)" "$(git rev-parse --short=7 HEAD)"`. The recommended output is monotonic `RELEASE.rREVISION`. The GUI should understand that `pkgver()` is executable code and that any PKGBUILD-parsing path must shell out to `makepkg --printsrcinfo` inside a sandboxed environment rather than `source`-ing the file directly, both for safety and because `startdir` semantics differ at different build phases.

## Checksum management

Checksum regeneration is two tools, one behavior. `makepkg -g` prints fresh integrity arrays to stdout in the algorithm chosen by `INTEGRITY_CHECK` in `/etc/makepkg.conf` (default `sha256`) and can be appended with `makepkg -g >> PKGBUILD`, though that creates duplicate arrays that require manual cleanup. `updpkgsums` from `pacman-contrib` is the correct primary tool: it downloads all sources, computes new hashes, and rewrites the existing `*sums*=()` arrays in place, preserving algorithm and location. The GUI should expose a "Regenerate checksums" button that invokes `updpkgsums --nocolor` and diffs the before/after per source entry, flagging `SKIP` entries as intentionally unpinned rather than as errors, warning when the count of checksums no longer matches the count of source entries, and surfacing algorithm mismatches (e.g., mixed `sha256sums` and `md5sums` for the same source) as suggestions to standardize on `sha256sums`. A common mistake is changing `source=()` entries without regenerating checksums, which produces the `==> ERROR: One or more files did not pass the validity check` abort; the GUI should automatically detect modifications to the source URL field and flag the checksum as stale in the form view.

## Local build and test

The build-test loop is the largest runtime-UX surface. Raw `makepkg` offers a huge flag set — `-s` syncs build dependencies via pacman, `-i` installs the result with `pacman -U`, `-r` removes the build deps afterward, `-c` cleans `src/` and `pkg/` after success, `-C` is cleanbuild (removes `$srcdir` before building), `-f` overwrites an existing built package, `-L` tees per-function logs to `$LOGDEST`, `-e` (`--noextract`) reuses the existing `$srcdir` without re-extraction and is the primary retry flag, `-o` (`--nobuild`) downloads and extracts without compiling, `--noarchive` stops after `package()`, `--holdver` suppresses `pkgver()` for VCS builds, and `--skipchecksums`, `--skippgpcheck`, and `--skipinteg` bypass integrity. Directory semantics are that `$srcdir` is where extraction happens (`$BUILDDIR/<pkgbase>/src/` when `BUILDDIR` is set, otherwise inside the invocation dir), `$pkgdir` is the staged root for `package()` (`$BUILDDIR/<pkgbase>/pkg/<pkgname>/`), `$startdir` is the PKGBUILD directory and is deprecated for use in functions. `makepkg.conf` is searched in order `/etc/makepkg.conf`, then `/etc/makepkg.conf.d/*.conf`, then `$XDG_CONFIG_HOME/pacman/makepkg.conf`, then `~/.makepkg.conf`, then a `--config` override, then environment variables (`PKGDEST`, `SRCDEST`, `SRCPKGDEST`, `LOGDEST`, `BUILDDIR`, `PKGEXT`, `GPGKEY`, `PACKAGER`). The GUI should expose these as a settings panel with per-package overrides, and it must ensure that `PACKAGER` is set because downstream tooling (including `checkpkg` and `.PKGINFO` inspection) relies on it.

The GUI should stream build output through a pty (not a pipe) — the build produces colored output and sometimes uses carriage returns, which only a VTE-backed widget renders correctly. In GTK the right primitive is `VteTerminal` from `libvte-gtk4`; in Qt it is `qtermwidget` (or a `QProcess`-backed pager with ANSI rendering); in web-based stacks it is `xterm.js` with `node-pty`. The GUI should invoke `makepkg` with `--nocolor` only if it is rendering its own colors from the log stream, and it should always pass `-L` so per-function logs land in `$LOGDEST` and can be reopened after the fact. Output parsing keys off `makepkg`'s message helpers: `==>` introduces a major stage (`Making package`, `Checking runtime dependencies`, `Retrieving sources`, `Validating source files with sha256sums`, `Verifying source file signatures with gpg`, `Extracting sources`, `Starting prepare()`, `Starting build()`, `Starting check()`, `Entering fakeroot environment`, `Starting package()`, `Tidying install`, `Creating package`, `Generating .PKGINFO/.BUILDINFO/.MTREE`, `Compressing package`, `Finished making`), `  ->` introduces a sub-step, `==> WARNING:` is a warning, and `==> ERROR:` precedes an abort. Per-source status lines end with `Passed`, `FAILED`, `Skipped`, or `NOT FOUND`. Exit codes are imperfectly stable but generally 0 is success, 4 signals a `build()` failure, 7 means pacman could not install dependencies, 11 means integrity failure, 12 means download failure, 13 means a package has already been built, and 15 means `pacman -U` failed — the GUI should surface both the numeric code and the matched text pattern when narrating errors. Common failures to render as first-class recoverable states are `Could not download sources` (offer retry, edit URL, or supply a local file), `did not pass the validity check` (offer `updpkgsums` then diff), `One or more PGP signatures could not be verified` (parse the keyid from the log and offer `gpg --recv-keys <id>`, then add to `validpgpkeys`), `Running makepkg as root is not allowed` (block with a clear explanation), and `Part of the package group has already been built` (offer `-f`).

Building on the user's own host is insufficient for a maintainer — it produces packages that appear to work only because the maintainer already has extra libraries installed. The correct pre-push test is a **clean chroot build** using devtools. The classic path is `mkarchroot /var/lib/archbuild/extra-x86_64/root base-devel` to create the pristine root once, then `makechrootpkg -c -r /var/lib/archbuild/extra-x86_64` from the PKGBUILD directory to build inside a fresh working-copy snapshot. The convenience wrapper `extra-x86_64-build` bundles both steps. The modern unified entry point is `pkgctl build`, which automatically chooses the correct repo-and-arch chroot, defaults to running namcap and checkpkg, supports `--repo`, `--arch`, `-c` for clean, `-s` for an interactive shell in the chroot, `-I pkg.pkg.tar.zst` for stacking a dependency into the chroot, `-i failure` to drop into an inspection shell on failure, and `--offload` for remote builders. The GUI should present a single primary "Build in clean chroot" button that wraps `pkgctl build --checkpkg --namcap` and expose the underlying flags through an advanced dialog. It must communicate that chroot builds require root (systemd-nspawn) and will prompt for sudo, that each pristine root consumes roughly 1 GB and each working copy another 1–5 GB, that Btrfs makes working copies essentially free via snapshots while ext4 or xfs performs full copies, and that `/var/lib/archbuild/<repo>-<arch>/<user>` directories need occasional cleanup. For users with constrained disk space the GUI should surface the tmpfs pattern of `mount -t tmpfs -o size=20G tmpfs /mnt/chroots/arch; extra-x86_64-build -r /mnt/chroots/arch`.

After the build, the GUI should run `namcap` twice. First on the PKGBUILD (`namcap PKGBUILD`) to catch missing `package()`, `pkgname-in-description`, `missing-license`, and redundancy with `base-devel`, and second on every `*.pkg.tar.zst` in `$PKGDEST` (`namcap *.pkg.tar.zst`) to catch `dependency-detected-not-included` (a shared library needed by an ELF in the package but not declared in `depends`), `dependency-included-but-already-satisfied` (redundant transitive), `dependency-not-needed` (declared but unused), `non-fhs-*`, `elf-file-outside-valid-path`, `cross-directory-hardlink`, `setuid-binary`, and several others. The tool's output format is `<pkgname> <LEVEL>: <message>` where level is `E:`, `W:`, or `I:`, and `namcap -m` switches to machine-parseable tags that the GUI should parse with a regex of `^(\S+)\s+([EWI]):\s+(\S+)(?:\s+(.*))?$`, mapping each tag to a tooltip string from `/usr/share/namcap/namcap-tags`. The GUI should render results grouped by severity with counts, let the user click each finding to jump to the relevant line (in the PKGBUILD) or file (in the tarball), and expose a per-project exclusion list that gets passed to subsequent runs as `namcap -e tag1,tag2`. Critically, the GUI must educate the user that namcap's ldd-based detection misses Python/Perl/shell runtime imports and plugins loaded via `dlopen()`, so a clean namcap run does not guarantee complete dependencies — the container smoke test complements it. Running `shellcheck --shell=bash -x PKGBUILD` alongside namcap catches many scripting issues that namcap does not; common PKGBUILD-specific false positives (SC2034 unused-variable for the canonical metadata fields, SC2164 `cd` without `||`) should be suppressed by default.

The GUI should also offer a container smoke test as a final gate. Using podman (rootless, preferred) or docker, the test is `podman run --rm -v "$PWD":/pkg:ro,Z archlinux:latest bash -c 'pacman -Syu --noconfirm && pacman -U --noconfirm /pkg/*.pkg.tar.zst && <entry-binary> --version'`. This catches missing runtime dependencies that even clean chroot builds miss (because the chroot contains the makedeps, some of which linger at runtime). A "minimal dependency test" variant installs only the packages declared in `depends=()` plus the built package and runs the entry binary — if it fails with a missing shared library, `depends` is incomplete. The GUI should render pass/fail per test and quote the tail of any failing output for quick diagnosis.

## Dependency validation

Declared dependencies should be validated against both official repositories and the AUR. Official repo lookups use `pacman -Si <pkg>` (not found returns non-zero and a recognizable error); AUR lookups use `GET https://aur.archlinux.org/rpc/v5/info?arg[]=<pkg>` which returns JSON with a `results` array whose members carry `Name`, `PackageBase`, `Version`, `Maintainer` (null when orphaned), `CoMaintainers`, `OutOfDate` (null or unix timestamp), `LastModified`, `NumVotes`, `Popularity`, full `Depends`/`MakeDepends`/`OptDepends`/`CheckDepends`, `Provides`, `Conflicts`, `Replaces`, `License`, and `Keywords`. Queries greater than roughly 200 packages risk the nginx URI limit of about 4443 bytes, so the GUI should chunk. Rate limiting is 4000 requests/day/IP, and the correct scaling strategy is to cache the hourly `https://aur.archlinux.org/packages-meta-ext-v1.json.gz` (full multiinfo dump, supports `ETag` and `Last-Modified`) locally and fall back to per-package RPC calls only for freshness checks on specific packages. The GUI's dependency form should therefore show, per entry, one of four states: "found in official repos," "found in AUR (link to package page)," "virtual/provided by <pkg>," or "not found" (red). Detecting *missing* dependencies — ones the maintainer forgot — is harder and relies on namcap's ELF scanning (`dependency-detected-not-included` with the specific library and owning package named in the tag parameters), supplemented by manually running `ldd` on binaries in `$pkgdir` inside the clean chroot and checking ownership with `pacman -Qo`. For library packages `find-libdeps foo.pkg.tar.zst` and `find-libprovides` produce pacman's `%DEPENDS%` and `%PROVIDES%` representations respectively, and `sogrep libfoo.so.X` lists packages linking against a soname (useful for planning rebuild cascades). `checkpkg` (devtools) downloads the previously-built version via `pacman -Spdd --noconfirm <pkg>` and diffs file lists and sonames side by side; it is the canonical soname-break detector and its output of `Sonames differ:` should be surfaced in a dedicated panel.

## .SRCINFO generation and synchronization

`.SRCINFO` is produced by `makepkg --printsrcinfo > .SRCINFO` and is **mandatory in every commit pushed to the AUR**. Its format is line-oriented `key = value` text with exactly one `pkgbase = ...` header and one or more `pkgname = ...` headers, emitting one line per array element and using per-arch suffixed keys where applicable. Out-of-sync `.SRCINFO` is the most common AUR packaging bug: any PKGBUILD metadata edit (`pkgver`, `pkgrel`, `depends`, `source`, checksums) invalidates it, and VCS `pkgver()` auto-bumps during a build do *not* propagate to `.SRCINFO`. The GUI should treat `.SRCINFO` as a derived artifact that must be regenerated and diffed visibly before any commit. Detection is simple: shell out to `makepkg --printsrcinfo` into a temporary buffer and diff against the on-disk file byte-for-byte; if they differ, surface a prominent banner with a one-click "Sync .SRCINFO" action. A belt-and-braces approach is to install a per-repository pre-commit hook that automatically regenerates `.SRCINFO` whenever PKGBUILD is in the staged set, mirroring what `aurpublish` does. The GUI should also offer a Git-style unified diff view of the old versus new `.SRCINFO` so the user can confirm that the changes are what they expected — a case where the metadata diff is more informative than the PKGBUILD diff, because it shows exactly what the AUR web-facing view will look like.

## Git workflow for AUR

The files that belong in the repository are `PKGBUILD`, `.SRCINFO`, and whatever `install=`, patch, `.service`, `.sysusers`, `.tmpfiles`, or `LICENSE` files the PKGBUILD references, plus optionally `.gitignore` and a `REUSE.toml`. Everything else is derived. The recommended `.gitignore` uses a whitelist pattern — ignore everything, then re-allow `.gitignore`, `.SRCINFO`, `PKGBUILD`, `*.install`, `*.patch`, `*.diff`, `*.service`, and `LICENSE` — requiring `git add -f` for new files, which is the safest default because it prevents accidental commits of `src/`, `pkg/`, or `*.pkg.tar.zst`. A denylist is acceptable and more conventional; it must at minimum exclude `src/`, `pkg/`, `*.pkg.tar*`, `*.src.tar*`, `*.sig`, and `*.log`.

Commit message conventions are informal but consistent: `Initial commit` for a first push, `Update to 1.2.3` or `upgpkg: 1.2.3-1` for version bumps, `pkgrel bump` for rebuild-only changes, `Rebuild for <lib> 2.0` for soname cascades, and descriptive imperative-mood messages for bug fixes. Commit identity is taken from `git config user.name` and `user.email`; pushed commits are effectively immutable (the AUR is append-only), so a mistake in identity requires users to live with it or delete and resubmit the package. Each commit must be self-consistent — `PKGBUILD` and `.SRCINFO` together in the same commit — because the server-side hook validates every commit in the pushed range, not only the tip.

Branch enforcement is the third mistake that catches new users. **AUR accepts pushes only to `master`**. Any other branch name, including the modern default `main`, is rejected with `pushing to a branch other than master is restricted`. The GUI should detect the current branch on repo open and offer to rename it with `git branch -m master` plus a suggestion to set `git config --global init.defaultBranch master` for future clones created by the tool. Force pushes are also rejected server-side — the AUR denies non-fast-forward updates — so rebases that rewrite already-pushed commits are impossible without deleting and recreating the package via a deletion request, and the GUI should block force-push actions with an explanation rather than let users try. Size limits are worth exposing: individual blobs are capped at 500 KiB (raised from 256 KiB in December 2020), any violation produces `remote: error: maximum blob size (500.00KiB) exceeded`, and source URLs in `source=()` are capped at 8000 characters each. Large upstream tarballs must be fetched by `source=()` at build time, never committed.

Before push, the GUI should present a review pane showing `git log --stat origin/master..HEAD`, a combined unified diff of queued commits, the diff between `makepkg --printsrcinfo` output and the committed `.SRCINFO` (catching the "forgot to regenerate" class of bug), the current `namcap PKGBUILD` output, and a reminder of the maintainer header convention ("Maintainer: Name <email>" on the first line, with previous maintainers moved to `# Contributor:` lines).

## Publishing to AUR

The push is `git push -u origin master` for the first push of a new namespace and `git push` for updates. Success produces the usual git enumeration and a `[new branch]` or fast-forward line ending with `master -> master`. Failures are surfaced by the server-side `aurweb-git-update` hook as `remote: error:` lines followed by `hook declined to update refs/heads/master` and `[remote rejected] master -> master (hook declined)`. The specific error strings to map to actionable recovery are `missing .SRCINFO` (some commit in the pushed range lacks the file — the GUI should offer interactive rebase assistance to apply `.SRCINFO` to earlier commits, or for the common single-commit case, `git commit --amend` after `git add .SRCINFO`), `missing PKGBUILD` (analogous), `invalid pkgbase: <name>, expected <name>` (the `.SRCINFO` pkgbase does not match the repo name; fix `pkgbase=` in PKGBUILD and regenerate), `missing pkgname entry` (`.SRCINFO` has no `pkgname =` lines — usually a regeneration bug or empty `pkgname` array), `missing install file: <name>.install` (PKGBUILD declares `install=` but the file is not committed — `git add -f` it), `pushing to a branch other than master is restricted` (rename branch), `denying non-fast-forward` (rebase attempted on already-pushed history — blocked, require deletion-and-recreate), `maximum blob size (500.00KiB) exceeded` (a committed file is too large; offer `git filter-repo` rewrite guidance), `source entry too long` (URL > 8000 chars), and `invalid pkgbase: <name>` without "expected" (illegal characters — the GUI should have prevented this upstream but must also recognize it). Separately, SSH-level `Permission denied (publickey).` indicates an auth problem rather than a hook rejection — the key is not attached, the user is not a maintainer or co-maintainer of an existing package, or `IdentitiesOnly` is not set.

After a successful push the package is immediately live at `https://aur.archlinux.org/packages/<pkgbase>`, raw files are browsable at `https://aur.archlinux.org/cgit/aur.git/tree/PKGBUILD?h=<pkgbase>` and the `.SRCINFO` at the analogous URL, and a snapshot tarball is at `https://aur.archlinux.org/cgit/aur.git/snapshot/<pkgbase>.tar.gz`. The GUI should open the package page in the default browser on push success and also fetch the RPC info endpoint to confirm that the new version is indexed (there can be a short lag for the list-index caches, but the package page itself updates within seconds). For new packages the user becomes the initial Maintainer automatically; no review is required.

## Long-term maintenance

Maintenance spans upstream tracking, pkgver and pkgrel hygiene, AUR web actions, and co-maintainer management. The GUI should integrate `nvchecker` as a background service: for each managed package it maintains a `nvchecker.toml` section keyed on pkgname, with a `source =` of `github`/`gitlab`/`pypi`/`aur`/`archpkg`/`gitea`/`regex`/`git` and the appropriate per-source options (e.g. `github = "owner/repo"`, `use_latest_release = true`, `prefix = "v"` for the common GitHub case; `pypi = "name"` for PyPI; `regex = "..."` with `url=` for custom pages). Running `nvchecker -c config.toml --logger=json` emits a line-delimited JSON stream with events `updated`, `up-to-date`, `no-result`, and `error` plus fields `name`, `version`, and `old_version` — the primary integration point for GUI dashboards. The companion tools are `nvcmp` for diffing `oldver.json` against `newver.json` and `nvtake` for promoting specific entries after the user has acted on the bump (a one-click "Accept update" in the GUI maps to `nvtake -c config.toml <pkg>`). `pkgctl version check/setup/upgrade` wraps the same machinery for Arch official packages and can be reused for AUR packages to some extent.

When upstream bumps a version, the correct action is to update `pkgver=`, **reset `pkgrel=1`**, regenerate checksums, regenerate `.SRCINFO`, build, commit, and push. When only the PKGBUILD itself changes (a new dependency, a patch, a flag fix, a rebuild against a soname-bumped library), `pkgver` stays the same and `pkgrel` increments to 2, 3, and so on. For VCS packages the `pkgver()` function output is the authority, so a PKGBUILD-only change that leaves `pkgver()` output unchanged still bumps `pkgrel`. Epoch is an escape valve of last resort — it prefixes the displayed version as `epoch:pkgver-pkgrel` and forces pacman's `vercmp` to see the package as newer regardless of the `pkgver` comparison; it is used only when upstream regresses version ordering, for instance switching from a date-based scheme like `20231231` back to semver `1.0`, or dropping a `v` prefix in a way that breaks ordering. The GUI should offer a "Bump version" action that auto-resets `pkgrel`, a "Bump pkgrel only" action that preserves `pkgver`, and a "Bump epoch" action guarded by an explanation and confirmation.

AUR web actions are browser-driven form posts; there is no official REST API for flag/unflag/request/co-maintainer. The URL patterns are `https://aur.archlinux.org/pkgbase/<pkgbase>/flag/` for flag-out-of-date with a `comments=` reason, `/unflag/` for maintainer-or-PM unflag, `/request/` with `type=orphan|deletion|merge` and `merge_into=` for orphan/deletion/merge requests, `/adopt/` and `/disown/` for orphan adoption and voluntary disowning, `/comaintainers/` with a newline-separated list of usernames for co-maintainer management, `/vote/` and `/unvote/`, `/notify/` and `/unnotify/` for email notifications, and POSTs with `comment=` for comments. Each request requires the session cookie `AURSID` and a CSRF `token` scraped from the preceding GET. The GUI can expose these actions by either opening the relevant URL in the default browser (safest, always works) or by automating the form post with credentials the user has explicitly granted (libsecret/kwallet-stored AUR session cookie). Because these endpoints are not a stable API and have changed shape before (most recently at AUR v6 in 2022), the GUI's automation layer should be feature-flagged and defer to browser-open when a form post fails. Orphan requests are auto-accepted if the package has been flagged out-of-date for 180 or more days; deletion requests go to the PM queue and may or may not be honored; merge requests are the appropriate form when a rename occurs. Co-maintainers have full push access; the GUI should surface the co-maintainer list for every managed package on the dashboard.

## Advanced cases and edge conditions

Split packages have been covered above but one detail deserves emphasis: the per-split `depends` is not consulted for `--syncdeps` during the build, so every runtime dependency of any subpackage must appear either in global `depends` or in global `makedepends`, and the GUI should warn when it detects a per-split `depends` entry that is not globally represented. VCS fragment handling should be a first-class editor widget: a dropdown for `git+`/`svn+`/`hg+`/`bzr+`/`fossil+` prefix, a URL field, a fragment type (`branch`/`tag`/`commit`/`revision`) with a value field, and a "signed" toggle that both appends `?signed` and prompts for `validpgpkeys` entries. Git submodule handling post-CVE-2022-39253 requires explicit `prepare()` logic that runs `git -c protocol.file.allow=always submodule update` after rewriting submodule URLs to point at sibling `$srcdir` clones; the GUI should offer a template snippet for this pattern because upstream writeups miss it frequently. PGP verification requires 40-character uppercase hex fingerprints in `validpgpkeys`, uses the user's `~/.gnupg` keyring (not pacman-key's `/etc/pacman.d/gnupg`), and is bypassed with `--skippgpcheck` — the GUI should never silently bypass a PGP failure; instead it should parse the failing keyid from the log, offer `gpg --recv-keys <id>` with a confirmation step, then offer to add the fingerprint to `validpgpkeys` after verification succeeds.

Binary `-bin` packages follow a stable convention: suffix `-bin`, `arch=(x86_64)` (or per-arch with `source_x86_64=()` and `source_aarch64=()`), `options=(!strip !debug)` because upstream artifacts should not be rewritten, `provides=("$_pkgname=$pkgver") conflicts=("$_pkgname")` to cooperate with the source-built variant, no `build()`, and an install-only `package()`. Rust packages require `cargo` in `makedepends` (not `rust` or `rustup`), `--frozen`/`--locked` in `cargo fetch`/`build`/`install` for reproducibility, `CARGO_TARGET_DIR=target` to localize output, and `cargo install --no-track --root "$pkgdir/usr"` for pkgdir installation; offline chroot builds require `CARGO_HOME=$srcdir/.cargo` so crates vendor under `$srcdir` and `--frozen --offline` works; upstream Cargo.toml `lto = true` often requires `options=(!lto)` to avoid clashing with makepkg's LTO; and `cargo-auditable` can be wrapped around the build to embed an SBOM. Go packages use `GOFLAGS="-buildmode=pie -trimpath -ldflags=-linkmode=external -mod=readonly -modcacherw"` with `CGO_*FLAGS` exported from caller `*FLAGS` for hardening parity — and the GUI must warn that `-trimpath` and the current debug-package machinery are mutually exclusive, forcing `options=(!debug)`.

The `conflicts`, `provides`, and `replaces` triad is often misused. `provides` declares that this package supplies a name or virtual capability and may carry a version (`provides=('foo=1.2.3')` or `provides=('libfoo.so=3-64')` for soname-versioned library provisioning); `conflicts` declares mutual-exclusion either with another pkgname or with another package's `provides` key; `replaces` causes pacman's `-Syu` to silently replace the listed package with this one and is the right mechanism for renames and deprecations but the wrong mechanism for alternative variants. Two packages providing the same `provides` key are implicitly mutually exclusive. The canonical VCS-variant pattern is `foo-git` with `provides=(foo) conflicts=(foo)` and no `replaces`, which allows users to switch between the release and VCS variants without `-Syu` forcing them one way.

Patches should be added to `source=()` with `SKIP` checksums if they are local files, and applied in `prepare()` with `patch -Np1 -i "$srcdir/fix.patch"` (the `-N` flag making the operation idempotent on re-runs) — storing the `.patch` files alongside the PKGBUILD in the repository, which is always fine because AUR's 500 KiB blob limit accommodates all reasonable patch sizes.

## GUI-specific design considerations

A first-run wizard is the right UX for SSH setup, first clone, and initial PKGBUILD authoring — the stages are strictly sequential and each depends on the previous. For day-to-day work a sidebar master-detail layout is superior: a sortable, filterable package list on the left (columns for name, local version, upstream version as reported by nvchecker, AUR version, out-of-date flag with age, votes, popularity, last-push timestamp, and co-maintainers; color-coded green/yellow/red/gray for in-sync/upstream-ahead/flagged-OOD-or-broken/orphaned); and a tabbed detail view on the right (tabs for Metadata form, Sources table, Dependencies, Build log, Preview diff, AUR web actions, nvchecker config). The libadwaita `AdwNavigationSplitView` or Qt `QSplitter` pattern is the natural fit. Real-time build output embeds through a pty-backed terminal widget (`VteTerminal` for GTK, `qtermwidget` for Qt, `xterm.js` with `node-pty` for Electron), with regex-driven error-line highlighting mapped to the `==>` event parser described earlier. Diff viewing should handle PKGBUILD, `.SRCINFO`, and patch files via `GtkSourceView` with a diff language or a Kate/KTextEditor embed, with a three-way merge fallback (`meld`, `kdiff3`) for the rare case of local-vs-upstream-vs-AUR divergence.

State persistence should live under XDG: `$XDG_CONFIG_HOME/<app>/` for human-readable TOML preferences (default build options, AUR username, editor theme), `$XDG_DATA_HOME/<app>/` for SQLite runtime state (managed-packages table, nvchecker history, build-log index, last-known AUR metadata, per-package notes), `$XDG_CACHE_HOME/<app>/` for transient blobs (downloaded `packages-meta-ext-v1.json.gz`, snapshot tarballs), and `libsecret` or `QtKeychain` for secrets (AUR session cookies, GitHub tokens, gpg passphrases). Working trees should live at `$XDG_DATA_HOME/<app>/pkgs/<pkgbase>/` by default, with user overrides per-package. Multi-package management is the primary value proposition over hand-rolled shell workflow: batch "check all" against nvchecker, batch "build selected in chroot," batch "push selected after dry-run," saved filters like "OOD > 30 days" or "upstream ahead" or "Rust only." Integration with the AUR RPC v5 info endpoint should be cached (ETag-aware, 24-hour TTL or on-demand refresh) and surface fields like OutOfDate timestamp, Maintainer/CoMaintainers/Submitter, NumVotes, Popularity, FirstSubmitted, and LastModified on the package detail view.

Notifications for upstream version changes should run as a user systemd service invoking nvchecker on a schedule (hourly or daily), emit desktop notifications via libnotify with action buttons for "Review in <app>" / "Bump now" / "Snooze 1 day" / "Mark ignored," and use a DBusActivatable `.desktop` entry so clicking the notification opens the GUI scrolled to the relevant package. Error recovery is the single biggest UX differentiator over CLI workflows: every failure modality should have a categorized recovery action. A build failure parses the log, classifies it (missing dep, checksum mismatch, PGP failure, network issue, `pkgver()` failure, compile error), quotes the relevant tail, and offers a primary action (install the missing dep, run `updpkgsums`, import the GPG key, retry with `-e`, drop into a chroot shell). A push rejection parses the hook error, maps to the relevant remedy (`git commit --amend` for `.SRCINFO`/PKGBUILD issues, branch rename for non-master, `git add -f` for missing install files, deletion-request guidance for force-push or rename cases), and never offers force-push because the server rejects it. Metadata edits in the form view should be undo/redo-aware at the field level, not just the textarea level, so Ctrl+Z restores the prior value rather than one character at a time, and auto-save drafts should land in SQLite so a crash never loses unsaved work.

## Gaps frequently missed in first implementations

Several subtle requirements show up repeatedly as bugs in greenfield AUR tooling and should be verified against the in-progress implementation. `.SRCINFO` must be regenerated and committed in **every** commit — not only the final one — because the server-side hook walks the full pushed range and rejects the first commit it finds without a valid `.SRCINFO`. Branch enforcement is hard: local `main` is the default in modern git and must be renamed to `master` explicitly; `git config --global init.defaultBranch master` is the cleanest prevention. SSH fingerprint verification against the three published SHA256 values (Ed25519, ECDSA, RSA) must happen before any push is attempted and must be refreshable from the live AUR homepage and ArchWiki, not hard-coded forever. VCS packages need a `pkgver()` function and `SKIP` checksums; forgetting either is caught by namcap or makepkg respectively but the GUI should warn at edit time. Split packages require `pkgbase` to match the AUR repository name exactly, and the regex `^[a-z0-9][a-z0-9.+_-]*$` must be enforced at input time to catch uppercase/prefix/dot violations before push. Licenses must migrate to SPDX; legacy `GPL`/`custom` should be flagged and one-click-converted. Epoch must never be bumped casually — doing so on a package in widespread use causes downstream pain, and the GUI should require confirmation with an explanation of `vercmp` semantics. `pkgver()` code runs on every build and can do anything; safe metadata parsing must go through `makepkg --printsrcinfo` inside a sandbox, never by `source`ing the PKGBUILD directly. Force-push is not available; any UX suggesting rebase-and-force on pushed history is wrong and must be replaced by deletion-request-and-re-push guidance. The AUR is unmoderated, so the push button is the commitment point and pre-push linting must be comprehensive: namcap both on PKGBUILD and on the built tarball, clean chroot build (not host build), container smoke test with minimal declared dependencies, `.SRCINFO` diff against the regenerated output, and a review of the pre-push commit range. Large blobs (above 500 KiB) and over-long source URLs (above 8000 characters) must be caught client-side with clear explanations. The `Maintainer:` and `Contributor:` header convention in the PKGBUILD first lines is not enforced by any tool but is a strong social norm and should be populated and preserved automatically, transitioning the previous maintainer to `# Contributor:` when the current user takes over a package.

## What to build next

The implementation should prioritize, in order, the SSH wizard with fingerprint verification and `IdentitiesOnly` handling, the structured PKGBUILD editor with SPDX license validation and VCS-fragment widget, the `.SRCINFO` auto-sync with diff, the clean-chroot `pkgctl build` integration with pty-embedded output, the namcap-plus-shellcheck linter pane with tag-to-prose mapping, the `updpkgsums` diff flow, the pre-push review dialog with commit range and error-to-remedy mapping, the AUR RPC cache backed by the hourly archive, and the nvchecker background service with libnotify notifications. The four highest-leverage UX patterns to nail are the sidebar package dashboard with color-coded status, the pty-embedded terminal with regex error mapping, the three-pane diff view (PKGBUILD / `.SRCINFO` / upstream-tag), and the error-recovery modal that categorizes every failure and offers a specific remedy rather than dumping a log. If those land well, the tool displaces the ad-hoc shell workflow that every AUR maintainer currently stitches together — which, as the survey of existing tools confirms, is still the state of the art despite aurpublish, aurutils, pkgbuild-mode, and a dozen abandoned half-starts, leaving this genuinely greenfield.
